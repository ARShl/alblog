# 你不知道的 JS 读书笔记 8

  虽然在执行的层级上，我们的大脑是以异步事件方式运作的，但我们的任务计划似乎还是以顺序，同步的方式进行：“你要先去商店，然后买件衣服，然后自拍很多照片”。这个过程看起来并不怎么符合异步事件方式。我们认真思考的时候很少是以事件的形式进行的。取而代之的是，我们按照顺序 A 然后 B,然后 C 这样计划着，并且会假定有某种形式的临时阻塞来保证.会等待 A 完成，C 会等待 B 完成。

  同步代码，语句是一条一套执行的，例如冒泡中临时变量，z=x;x=y;y=z;这三条语句是同步执行的，这三条语句临时绑定按照顺序一个接一个地执行。

  如果说同步的大脑能够很好地映射到同步语句，那么在异步代码又是怎样的呢？你要去商店，但是路上接到电话，“去买件卫衣吧。xx 商店有卖。”你在手机地图上查看商店的地址，但是信号弱需要加载，你意识到自己忘了穿外套，外面有点冷，然后继续走。这时候路口红灯亮起了，地图终于加载出来了。于是...

  这样计划一天要做什么按什么顺序来的话，就会不清楚。但是。在大脑执行方面，他就是这么做的。不是多任务，而是快速的上下文切换。

  对于我们程序员来说，编写异步代码，假如回调是唯一实现手段时，困难之处就在于这种思考/计划的意识流对我们大多数人来是不自然的。我们的思考方式是一步一步的，但是从同步转换到异步之后，可用的工具（回调）却不是按照一步一步的方式表达的。

listen('click', function handler(evt) {
  setTimeout(function request() {
    ajax('url', function response(text) {
      if (text == 'hello') {
        handler()
      } else if (text == 'world') {
        request()
      }
    }),
      500
  })
})

doA(function () {
  doB()
  doC(function () {
    doD()
  })
  doE()
})

doF()

  这种就是常见的回调地狱（callback hell)。这个执行的顺序是 doA,doF,doB,doC,doE,doD.真实的异步 js 程序代码要混乱的多，追踪的难度会成倍增加。我们的大脑能够以顺序的方式（这个，这个，然后这个）计划一系列任务，但大脑运作的事件化的本质使控制流的恢复/重试/复制几乎不费什么力气。你去商店的时候发现手机拉家里了。那么你不会因为这一点就崩溃了。你很容易针对这个意外做出计划：回家拿手机，然后再去商店好了。但是代码上我们做不到也没有办法来维护了。

//A
ajax('url',function(..){
  //c
})

//B

  a 和 b 发生在现在，在 js 主程序的直接控制之下。而 c 会延迟到将来发生，并且在第三方控制下，ajax。基本上，这种控制的转移不会给程序带来很多问题。但很多时候 ajax 也就是你回调的不在你控制下，他是第三方提供的工具方法。这称为控制反转。也就是把自己程序一部分的执行控制交给某个第三方。

  这个涉及到信任问题。假如你在为一个销售昂贵的奢侈品做商务结账系统，你已经做好了结账系统的各个洁面。在最后一页，当用户点击确定就可以购买电视时，你需要调用数据统计公司提供的第三方函数以跟踪这个交易。可能为了提高性能，他们提供了一个看似用于异步追踪的工具，这意味你需要传入一个回调函数。在传入的 continuation 中，你需要提供向客户收费和展示感谢页面的最终代码。

shence.track(purchase, function () {
  chargeCreditCard()
  displayThankyouPage()
})

var tracked = false
shence.track(purchase, function () {
  if (!tracked) {
    tracked = true
    chargeCreditCard()
    displayThankyouPage()
  }
})

  这样没问题哦。某个上班之前的早晨，你像往常一样在楼下包子铺买包子。突然，老板惊慌失措地打电话过来，让你丢下包子赶紧到办公室。到了办公室，你得知一个客户买了一台电视，信用卡却被刷了 5 次，他很生气，这可以理解，客服已经道歉并启动了退款流程。但是。老板需要知道这样的事情为何出现。“这种情况你都没有测试过吗？！”。你甚至都不记得自己写过这段代码。通过分析日志，得出一个结论，唯一的解释就是那个分析工具出于某种原因把你的回调调用了五次而不是一次。然后你去找提供接口的第三方，他们说做了一些试验性代码，在某种情况下，会多次调用回调函数。他们很是尴尬，充满了歉意。保证这样的事情不会再发生，等等。

  你和老板讨论这件事，他对这个解释不满意。他认为，你不能在信任他们了（你们受到了伤害）。这时候你会怎么做，第一反应是什么？跟老板说。我们又管不了别人怎么做吗？你需要找到某种方法来保护结账代码，保证不再会出问题。

  这时候 QA 工程师问道：如果他们根本不调用这个回调怎么办？这个时候你就考虑他们调用你回调时所有可能出错情况。粗略列一下可能想到的问题：调用回调过早，回调过完没有回调。调用次数过多或过少。没有把需要的参数传给你的回调函数。吞掉可能出现的错误或异常。这感觉就像是一个麻烦列表。你可能开始意识到，对于被传给你无法信任工具的回调，你都不得不创建大量的混乱逻辑。这也是回调地狱的可怕之处。

function plus(x, y) {
  return x + y
}

plus(21, 21) //42
plus(21, '21') //2121

function plus(x, y) {
  x = Number(x)
  y = Number(y)

  return x + y
}

  不只是是别人的代码。大体上说，这等价于那条地缘政治原则，信任，但要核实。所以你的代码中使用了回调，尤其是但也不限于第三方工具，而且你还没有某种逻辑来解决所有这些控制反转导致的信任问题。那你的代码现在已经有了 bug,即使他们还没有给你造成损害。隐藏 bug 也是 bug。确实是地狱。

  回调设计存在几个变体。意在解决一些信任问题。有些 api 设计提供了分离回调（一个用于成功通知，一个用于出错通知）。还有一种回调模式脚 error-first 风格。node.jsAPI 都采用这种风格。其中回调的第一个参数保留用作错误对象，如果成功的话，这个参数就会被清空/置假。不过，如果产生了错误结果，那么第一个参数就会被置真。

function success(){
  ...
}

function failure(){
  ...
}

ajax('url',success,failure)

function response(err,data){
  if(err){
    failure()
  }else{
    success()
  }
}
ajax('url',response)

  这两种情况并没有像表面看上去那么真正解决主要的信任问题。这并没有阻止或过滤不想要的重复调用回调的问题。现在事情更糟了，因为现在你可能同时得到失败和成功的结果。你还是不得不编码处理所有这些情况。另外，不要忽略这个事实：尽管这是一种你可以采用的标准模式，但是它肯定更加冗长和模式化，可复用性不高，所以你还得不厌其烦地给应用中每个回调函数添加这样的代码。

  QA 提出的完全不调用这个信任问题怎么解决呢。可能需要设置一个超时来取消事件。可以构造一个工具来帮助实现这一点：

function timeoutify(fn, delay) {
  var intv = setTimeout(function () {
    intv = null
    fn(new Error('Timeout'))
  }, delay)

  return function () {
    if (intv) {
      clearTimeout(intv)
      fn.apply(this, arguments)
    }
  }
}

function foo(err, data) {
  if (err) {
    console.error(err)
  } else {
    console.log(data)
  }
}

ajax('url', timeoutify(foo, 500))

  回调函数是js异步的基本单元，但是业务越来越成熟，回调已经不够用了。幸好ES6带着一些极好的答案来了。
