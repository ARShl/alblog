# 你不知道的 JS 读书笔记 7

  实际上，所有重要的程序都需要通过这样或那样的方法来管理这段时间间隙，这时可能是在等待用户输入，从数据库或文件系统中请求数据，通过网络发送数据并等待相应，或者是在以固定时间间隔执行重复任务（比如动画）。在诸如此类的场景中，程序都需要管理这段时间间隙的状态。

  可能你已经听说过，可以发送同步 Ajax 请求。尽管技术上说是这样，但是，在任何情况下都不应该使用这种方式，因为它会锁定浏览器 UI（按钮，彩蛋，滚动条等），并阻塞所有的用户交互。这是一个可怕的想法，一定要避免。为了避免回调函数引起的混乱并不足以成为使用阻塞式同步 AJax 的理由。

function now() {
  return 21
}

function later() {
  answer = answer * 2
  console.log('meaning of life:', answer)
}

var answer = now()

setTimeout(later, 1000) // meaning of life: 42

  这个程序有两个块：现在执行的部分，以及将来执行的部分。到底什么时候控制台 I/O 会延迟，甚至是否能够被观察到，这都是游移不定的。如果在调试的过程中遇到对象在 console.log(...)语句之后被修改，可你看到了意料之外的结果，要意识到这可能是这种 I/O 的异步化造成的。

  如果遇到这种少见的情况，最好的选择是在 js 调试器中使用断点，而不要依赖控制台输出。次优的方案是把对象序列化到一个字符串中，以强制执行一次快照，比如通过 JSON.stringify(...)

  js 引擎都提供了一种机制来处理程序中多个块的执行，且执行每块时调用 js 引擎，这种机制被称为事件循环。事件循环把自身的工作分成一个个任务并顺序执行，不允许对共享内存的秉性访问和修改。通过分立线程中彼此合作的事件循环，并行和顺序执行可以共存。并行线程的交替执行和异步事件的交替调度，其粒度是完全不同的。

  异步是关于现在和将来的时间间隙，而并行是关于能够同时发生的事情。并行计算最常见的工具就是进程和线程。进程和线程独立运行，并可能同时运行：在不同的处理器，甚至不同的计算机上，但多个线程能够共享单个进程的内存。

  js 从不跨线程共享数据，这意味着不需要考虑这一层次的不确定性。由于 js 的单线程特性，foo()开始运行，它的所有代码都会在 bar()中的任意代码运行之前完成，或者相反。这称为完整运行特性。

// 当进程中要处理超过1000万条记录时，页面上其他代码都不能运行。
// 可以异步地处理这些结果。每次处理之后返回事件循环，让其他等待事件有机会运行。

var res = []
function response(data) {
  // 一次处理1000个
  var chunk = data.splice(0, 1000)

  res = res.concat(
    chunk.map(function (val) {
      return val * 2
    })
  )

  if (data.length > 0) {
    // 异步调度下一批处理
    setTimeout(function () {
      response(data)
    }, 0)
  }
}

ajax('url1', response)
ajax('url2', response)

  setTimeout(...,0)(hack)进行异步调度，基本上它的意思就是“把这个函数插入到当前事件循环队列中的结尾处”。理论上说，人物循环可能无限循环，进而导致程序的饿死，无法转移到下一个事件循环 tick。从概念上看，这和代码中的无限循环（while(true))的体验几乎是一样的。

  有一个有趣的理论我想讲给你听。很多人说自己是能一心多用，比如左手画圆右手画方，边走路边嚼口香糖，边过马路边发微信。我们真的能同时执行两个有意识的，故意的动作，并对二者进行思考或推理吗？我们的大脑是以并行多线程的形式运行的吗？很可能并不是这样。我们更多是单任务执行者。实际上，在任何特定的时刻，我们只能思考一件事情。

  这里说的不是我们不自觉，无意识地自动完成的脑功能，比如心跳，呼吸和眨眼等。我们并不需要有意识地分配脑力来执行这些任务。

  我们讨论的是此时处于意识前端的那些任务。就像我在跟你讲这个东西，我还在执行任何其他更高级的脑功能吗？没有，我容易分心，在跟你讲的过程也在分心。

  我们假装并行执行了多个任务时，实际上极有可能是在进行快速的上下文切换，比如和朋友或家人电话聊天的时候还在打字。我们在两个或多个任务之间快速地来回切换，同时处理每个任务的微小片段，我们切换得如此之快，以至于对外界来说，我们就像是在并行地执行所有任务。 那这个和 js 的异步事件并发机制很相似。当然神经学没有这么简单了。但是这种方式有点类似于事件循环队列。如果把我打出来的每个词看作一个异步事件，那么在这一句中我的大脑有几十次机会被其他某个事件打断，比如我的感官甚至随机思绪。

  我不会每次可能呗打断的时候都转入其他进程中。但是中断的发生经常频繁到感觉自己的大脑几乎是在不停地切换到不同的上下文中。js 引擎可能也是这种感觉。

  如何表达和控制持续一段时间的程序行为。这不仅仅是指 for 循环开始到结束的过程，当然这也需要持续一段时间（几微秒）才能完成。它是指程序的一部分现在运行，而另一部分则在将来运行，现在和将来之间有段间隙，在这段间隙中，程序没有活跃执行。

  事实上，程序中现在运行的部分和将来运行的部分之间的关系就是异步编程的核心。可以把 js 程序写在单个.js 文件中，但是这个程序几乎一定是由多个块构成的，这些块中只有一个是现在执行，其他的则会在将来之行。最常见的单位是函数。经常遇到的一个问题是：程序中将来执行的部分并不一定在现在运行的部分立即执行完之后立即执行。换句话说，现在无法完成的任务将会异步完成，因此并不会出现人们本能地认为会出现的活希望的阻塞行为。

  常用的 ajax 请求，都是异步调用，ajax('url',function callback(res){res.data...}),虽然可以发送同步 ajax 请求，但是它会锁定浏览器的 ui,并阻塞所有的用户交互。任何时候，只要把一段代码包装成一个函数，并指定它在相应某个事件（定时器，鼠标点击，ajax 相应等）时执行，你就是在代码中创建了一个将来执行的块，也由此在这个程序中引入了一部机制。

  控制台是异步的，并没有什么规范或一组需求指定 console.*方法如何工作。它们并不是 js 正式的一部分，而是宿主环境添加到 js 中的。因此，不同的浏览器和 js 环境可以按照自己的意愿来实现，有时候会引起混淆。因为在某些条件下，某些浏览器的 console.log(...)并不会把传入的内容立即输出。出现这种情况的主要原因是，在许多程序中，I/O 是非常低速的阻塞部分。所以，从页面/ui 的角度来说，浏览器在后台异步处理控制台 I/O 能够提高性能，这时用户意识不到其发生。

var a = {
  index: 1
}

console.log(a) //??

a.index++

   这个其实是不确定的。如果在调试过程中遇到对象在 console.log(...)语句之后被修改，你看到了意料之外的结果，要意识到这可能是这种 I/O 的异步化造成的。如果遇到这种少见的情况，最好的选择是在 js 调试器中使用断点，而不要依赖控制台输出。次优的方案是把对象序列化到一个字符串中以强制执行一次快照，比如 JSON.stringify()

   我们来澄清一件事情，尽管能够编写异步的 js 代码，例如 settimeout,但直到 es6,js 才真正内建有直接的异步概念。js 引擎并不是独立运行的，它运行在宿主环境中，对于我们来说就是 web 浏览器，还有通过 node.js 去其他环境中，这些环境有一个共同点，即它们都提供了一种机制来处理程序中的多个块执行，且执行每块时调用 js 引擎，这种机制被称为事件循环。换句话说，js 引擎本身并没有时间的概念。只是一个按需执行 js 任意代码片段的环境。事件调度总是由包含它的环境进行。

  举例来说，如果你的 js 程序发出一个 ajax 请求，从服务器获取一些数据，那你就在一个回调函数中设置好相应代码，然后 js 引擎会通知宿主环境：现在我要暂停执行，你完成网络请求，拿到了数据，就请调用这个函数。然后浏览器就会设置侦听来自网络的相应，拿到药给你的数据之后，就会把回调函数插入事件循环，以此实现对这个回调的调度执行。

var eventLoop = []
var event

// 永远执行
while (true) {
  if (eventLoop.length > 0) {
    event = eventLoop.shift()

    try {
      event()
    } catch (err) {
      reportError(err)
    }
  }
}

  有一个用 while 循环实现的持续运行的循环，循环每一轮称为一个 tick。对每个 tick 而言。如果在队列中有等待时间，那么就会从队列中摘下一个事件并执行。这些事件就是你的回调函数。一定要清楚，setTimeout(...)并没有把你的回调函数挂在事件循环队列中，它所做的是设定一个定时器。当定时器到时后，环境会把你的回调函数放在事件循环中，这样，在未来某个时刻的 tick 会执行这个回调。如果这时候的事件循环中已经有 20 个项目了会怎么样呢，你的回调就会等待。它得排在其他项目后面。通常没有抢占式的方式支持直接将其排到队首。这也解释了为什么 setTimeout 的定时器精度可能不高。大体来说，只能保证你的回调函数不会在指定的时间间隔之前运行，但可能会在那个时刻运行，也可能在那之后运行，要根据事件队列的状态而定。

  术语异步和并行常常被混为一谈，但实际上它们的意义完全不同。记住，异步是关于现在和将来的时间间隙，而并行是关于能够同时发生的事情。并行计算最常见的工具就是进程和线程。进程和线程独立运行，并可能同时运行：在不同的处理器，甚至不同的计算机上，但多个线程能够共享单个进程的内存。与之相对的是，事件循环把自身的工作分成一个个任务并顺序执行，不允许对共享内存的并行访问和修改。通过分立线程中彼此合作的事件循环，并行和顺序执行可以共存。并行线程的交替执行和异步事件的交替调度，其粒度是完全不同的。

  js 是单线程运行的。多线程编程是非常复杂的。因为不通过特殊的步骤来防止中断和交错运行，会得到出乎意料，不确定的行为。由于 js 的单线程特性，foo(),bar()中的代码具有原子性，也就是说一旦 foo()开始运行，它的所有代码都会在 bar()中的任意代码运行之前完成。或者相反。这称为完整运行(run-to-completion)特性。

var a = 1
var b = 2
function foo() {
  a++
  b = b * a
  a = b + 3
}

function bar() {
  b--
  a = 8 + b
  b = a * 2
}

ajax('url', foo)
ajax('url', bar)

  由于 foo（）bar()不会被中断，所以程序只有两个可能的输出，取决于这两个函数哪个先运行。如果存在多线程，且 foo 和 bar 中的语句可以交替运行话，可能输出的数目会增加不少。

  底部加载更多。常见的情况是，触发了多次 ajax。返回的顺序假如不准确，可能会出现第 4 页的数据显示在第二页的位置。如果出现这样的交互，就需要对它们进行协调以避免竟态的出现。

var res = []
function response(data) {
  res.push(data)
}

function response(data) {
  if (data.url.page == 1) {
    res[1] = data
  } else if (data.url.page === 2) {
    res[2] = data
  }
}

ajax('url', response)
ajax('url', response)

  这种竟态我们可以协调下。这个例子中都会使 baz 过早运行。还有一种种优化叫做门闩。

var a, b
function foo(x) {
  a = x * 2
  baz()
}

function bar(y) {
  b = y * 2
  baz()
}

function bar(y) {
  b = y * 2
  if (a && b) {
    baz()
  }
}

function baz() {
  console.log(a + b)
}

ajax('url', response)
ajax('url', response)

var a, b
function foo(x) {
  a = x * 2
  baz()
}

function bar(y) {
  a = y * 2
  baz()
}

function bar(y) {
  if (!a) {
    b = y * 2
    baz()
  }
}

function baz() {
  console.log(a)
}

ajax('url', response)
ajax('url', response)

  还有一种并发合作方式。这里重点不再是通过共享作用域中的值进行交互。这里的目标是渠道一个长期运行的进程，并将其分割成多个步骤或多个任务。使其他并发有机会将自己的运算插入到事件循环队列中交替运行。比如处理1000万的数据。让别的事件也可以插件来做事。我们可以切割1000个执行，假如还有数据。我们通过settimeout异步继续执行，这时给其他等待事件有机会运行。这是一种hack.
