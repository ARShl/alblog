# 你不知道的 JS 读书笔记 9

  我们确定了回调表达程序异步和管理并发的两个主要缺陷：缺乏顺序性和可信任性。我们首先要解决的是控制反转问题，其中，信任很脆弱，也很容易失去。但是，如果我们能够把控制反转再反转回来。会怎么样呢？如果我们不把自己程序的 continuation 传给第三方，而是希望第三方给我们提供了解其任务何时结束的能力，然后由我们自己的代码决定下一步做什么，那将会怎样呢？这种范式就称为 Promise.

  在展示 promise 代码之前，先从概念上完整地解释 promise 到底是什么。设想一个这样的场景：去快餐店点了个汉堡，付款后收银员给了你一个有订单号的收据。订单号就是一个保证，保证你最终会得到汉堡。

  在等待的过程中，你可以做其他的事情，自拍。你把订单号当作芝士汉堡的占位符。从本质上讲，这个占位符使得这个值不再依赖时间。这是一个未来值。终于，当你听到服务员喊订单 113。然后愉快地拿着收据走到柜台，把收据交给收银员，换来了你的汉堡。换句话说，一旦我需要的值准备好了，你就用承诺值 value-promise 换取这个值本身。但是还有一种可能，他们叫到了你的订单号，但是，收银员告诉你，汉堡卖完了。所以。未来值可能成功也可能失败。

  在代码中，事情并非这么简单。这是因为，用类比的方式说就是，订单号永远不会被叫到。这种情况下，就永远处于一种未决议状态。

function add(getX, getY, cb) {
  var x, y
  getX(function (xVal) {
    x = xVal
    if (y != undefined) {
      cb(x + y)
    }
  })

  getY(function (yVal) {
    y = yVal
    if (x != undefined) {
      cb(x + y)
    }
  })
}

add(fecthX, fetchY, function (sum) {
  console.log(sum)
})

function add(xPromise, yPromise) {
  return Promise.all([xPromise, yPromise]).then(function (values) {
    return values[0] + values[1]
  })
}

add(fetchX(), fetchY()).then(function (sum) {
  console.log(sum)
})

  fetchX 和 fetchY 是直接调用的，他们的返回值 Promise 被传给 add.这些 promise 代表的底层值的可用时间可能是现在或将来，但不管怎么样，promise 归一保证了行为的一致性。第二层 add.通过 promise.all()创建并返回 promise。我们通过调用 then()等待这个 promise。

  就像买汉堡一样，Promise 的决议结果可能是拒绝而不是完成。拒绝值和完成的 Promise 不一样：完成值是编程给出的，而拒绝值，通常称为拒绝原因，可能是程序逻辑直接设置的，也可能是从运行异常隐式得出的值。

add(fetchX(), fetchY()).then(
  function (sum) {
    console.log(sum)
  },
  function (err) {
    console.error(err)
  }
)

  如果在获取 X 或 Y 的过程中出错，或者在加法过程中出错，add 返回的就是一个被拒绝的 promise,传给 then()的第二个错误处理回调就会从这个 promise 中得到拒绝值。从外部看，由于 promise 封装了依赖于时间的状态，等待底层值的完成或拒绝，所以 Promise 本身是与时间无关的。因此，Promise 可以按照可预测的方式组成，而不用关系时序或底层的结果。Promise 是一种封装和组合未来值的易于复用的机制。

  Promise 也是一种在异步任务中作为两个或更多步骤的流程控制机制，时序上的 this-then-that。使用回调，就是通知任务调用的回调。而使用 promise 的话，我们把这个关系反转了过来，侦听来自任务的事件，然后在得到通知的时候，根据情况继续。

function foo(x) {
  return listener
}

var evt = foo(42)

evt.on('completion', function () {
  // 完成进行下一步
})

evt.on('failure', function (err) {
  // 出错了。
})

var evt = foo(42)

//让bar(...)侦听foo(..)的完成
bar(evt)

// 让baz(...)侦听foo(..)的完成
baz(evt)

  foo 显式创建并返回了一个事件订阅对象，调用代码得到这个对象，并在其上注册了两个事件处理函数。相对于面向回调的代码，这里的反转是显而易见的，而且这也是有意为之的。这里没有把回调传给 foo(..),而是返回一个名为 evt 的事件注册对象，由它来接受回调。回调本身就表达了一种控制反转。所以回调模式的反转实际上对反转的反转，或者称为反控制反转-把控制返回给调用代码，这也是我们最开始想要的效果。

p.then(function(){
  p.then(function(){
    console.log('c')
  })
  console.log('a')
})

p.then(function(){
  console.log('b')
})
